Workflow Memory API

Memory for your workflows

FlowState gives automation tools a simple REST memory layer so each workflow run can remember what happened before.

Built for Zapier, n8n, Make, and any custom automation that needs lightweight state without standing up a database.

Start in 3 steps Open API Tester

Simple REST API

Store, recall, and forget values with predictable JSON responses that fit cleanly into HTTP nodes and webhook steps.

Automation Friendly

Designed for no-code and low-code tools that need persistence between workflow executions without extra infrastructure.

Predictable Contract

Every endpoint returns HTTP 200. Inspect success, message, and value in the JSON body to decide next steps.

How FlowState fits into a workflow

1
Write memory

Store a key when a workflow completes an important step.

2
Read memory later

Recall the key in the next run to avoid duplicate work or continue from the latest state.

3
Expire or delete

Use TTLs or explicit deletes to keep state fresh and scoped to the workflow lifetime.

Quick Start

From zero to workflow memory in three steps.

1

Get your API key

All endpoints except /v1/health require the X-API-Key header. Keys must be exactly 8 uppercase letters or digits.

To request an API key, contact the admin at shiv@flowstate.dev

Length: 8 Pattern: [A-Z0-9] Header: X-API-Key
2

Store a value

Send the key, value, TTL, and namespace. Namespaces are optional and default to _default.

curl -X POST https://your-domain/v1/store \
  -H "X-API-Key: K7M2X9BF" \
  -H "Content-Type: application/json" \
  -d '{"key": "last_run", "value": "2024-01-15T10:30:00Z", "ttlSeconds": 86400, "namespace": "invoice-sync"}'
3

Recall it later

Read the stored value in a future workflow run. Use the same key and namespace pair.

curl "https://your-domain/v1/recall?key=last_run&namespace=invoice-sync" \
  -H "X-API-Key: K7M2X9BF"

Important response behavior

FlowState always returns HTTP 200. Your workflow must check the JSON body and branch on success rather than HTTP status code.

POST

/v1/store

Persist a value for a key, optionally scoped by namespace and TTL.

Request body

keyRequired · 1-32 chars · [a-zA-Z0-9_-]
valueRequired · not null · max 1024 bytes
ttlSecondsOptional · 0-86400
namespaceOptional · max 32 chars · defaults to _default

Auth

Requires X-API-Key with exactly 8 characters matching [A-Z0-9].

Success contract

The body includes success and message. Your workflow should treat success: false as a handled failure even though HTTP 200 is returned.

curl -X POST https://your-domain/v1/store \
  -H "X-API-Key: K7M2X9BF" \
  -H "Content-Type: application/json" \
  -d '{"key": "last_run", "value": "2024-01-15T10:30:00Z", "ttlSeconds": 86400, "namespace": "invoice-sync"}'
{
  "success": true,
  "message": "Value stored successfully"
}
GET

/v1/recall

Fetch a value that was stored earlier.

Query parameters

keyRequired · same validation as store
namespaceOptional · defaults to _default

Response

When found, the response includes value. Missing or invalid requests still come back as HTTP 200 with success: false.

curl "https://your-domain/v1/recall?key=last_run&namespace=invoice-sync" \
  -H "X-API-Key: K7M2X9BF"
{
  "success": true,
  "message": "Value found",
  "value": "2024-01-15T10:30:00Z"
}
DELETE

/v1/forget

Delete a stored value when it is no longer needed.

Query parameters

keyRequired · same validation as store
namespaceOptional · defaults to _default

Typical use cases

Clear temporary locks, reset workflow checkpoints, or remove data after a workflow has safely handed it off elsewhere.

curl -X DELETE "https://your-domain/v1/forget?key=last_run&namespace=invoice-sync" \
  -H "X-API-Key: K7M2X9BF"
{
  "success": true,
  "message": "Value deleted successfully"
}
GET

/v1/health

Check service availability without authentication.

No auth required

This is the only public endpoint. Use it for uptime checks, smoke tests, or preflight health monitoring.

Automation tip

Even for health checks, build your workflow to read the response body rather than assuming non-200 means unhealthy.

curl https://your-domain/v1/health
{
  "success": true,
  "message": "FlowState is healthy"
}

Zapier Integration

Add memory to multi-step Zaps using Webhooks by Zapier — prevent duplicates, track progress, and share state between runs.

⚠️ Important: FlowState always returns HTTP 200 (even on errors). Use the success field in the response body — not the HTTP status — to drive Filter/Paths logic in your Zap.

Pattern 1 — Store & Recall (Checkpointing)

Store the last processed identifier or timestamp after a successful step, then recall it at the start of the next run to skip already-processed records.

1

Recall first

Add a Webhooks by Zapier → GET step against /v1/recall with your key and namespace in the query string.

2

Branch on success

Use a Filter or Paths step to inspect the JSON body: if success = true, use the stored value to skip already-processed items.

3

Store after completion

Add a POST webhook to /v1/store to update the key after the workflow finishes successfully.

Method: POST
URL: https://your-domain/v1/store
Headers:
  X-API-Key: YOUR_KEY
  Content-Type: application/json
Data:
{
  "key": "last_processed_invoice",
  "value": "{{123456__invoice_id}}",
  "ttlSeconds": 86400,
  "namespace": "zapier-invoice-sync"
}

Pattern 2 — Duplicate Detection (Deduplication)

Prevent duplicate execution when Zapier retries, webhooks fire twice, or users double-submit forms. Store a unique key before processing — if it already exists, skip.

1

Check if already processed

Recall a unique event key. If success = true → it's a duplicate, skip the rest.

2

Execute your action

If success = false (key not found), proceed with the action (create order, send email, etc.).

3

Mark as processed

Store the event key with a short value like "done" so future duplicates are detected.

# Step 1: Check if event was already processed
Method: GET
URL: https://your-domain/v1/recall?key=evt_stripe_1A2B3C&namespace=webhook-dedup
Headers:
  X-API-Key: YOUR_KEY

# If success=false → new event, proceed with action

# Step 3: Mark as processed
Method: POST
URL: https://your-domain/v1/store
Headers:
  X-API-Key: YOUR_KEY
  Content-Type: application/json
Data:
{
  "key": "evt_stripe_1A2B3C",
  "value": "processed",
  "ttlSeconds": 3600,
  "namespace": "webhook-dedup"
}

Real-World Zapier Use Cases

Webhook Events (Stripe, PayPal, Razorpay)

Payment providers may deliver the same webhook event multiple times. Use the event ID as your key to detect and skip duplicates.

key: evt_stripe_12345

Form Submissions

Users may double-click submit buttons or refresh the page. Combine user ID + form context as a dedup key.

key: user_789_lead_form

Order Processing

Prevent duplicate order creation when Zapier retries a step. Use the order ID as the key.

key: order_20260207_99

Email / Notification Guards

Send only one welcome email per user per day, or one alert per incident. Store the guard key with a TTL.

key: welcome_email_user42

Designing Good Keys

Choose a unique combination that identifies the specific operation instance:

✅ Good: evt_stripe_12345 — webhook event ID (unique per event)

✅ Good: user_789_create_order — user + action combination

✅ Good: invoice_sync_last_id — checkpoint cursor

❌ Avoid: 2026-02-07T09:53Z — timestamps are not unique across retries

❌ Avoid: generic keys like data or temp — collisions across workflows

Typical Zap Flow

┌─────────────────┐
│  Trigger fires  │  (webhook, schedule, form, etc.)
└────────┬────────┘
         ▼
┌─────────────────┐
│  GET /v1/recall │  Check if key exists
└────────┬────────┘
         ▼
    ┌────────────┐
    │success=true│──── YES ──→ SKIP (already processed)
    └────┬───────┘
         │ NO
         ▼
┌─────────────────┐
│  Execute Action │  (create order, send email, etc.)
└────────┬────────┘
         ▼
┌─────────────────┐
│ POST /v1/store  │  Mark key as processed
└─────────────────┘

n8n Integration

Use HTTP Request nodes to persist state between executions.

Recommended node setup

  • Use HTTP Request nodes for FlowState calls.
  • Set headers explicitly, including X-API-Key.
  • Read {{$json.success}} in IF nodes because the transport status is always 200.

Best fit scenarios

  • Checkpointing long-running workflows
  • Skipping records already processed
  • Sharing data across separate n8n workflows
URL: https://your-domain/v1/recall?key=cursor&namespace=n8n-sync
Method: GET
Headers:
  X-API-Key: {{$env.FLOWSTATE_API_KEY}}

Then use an IF node:
{{$json.success}} equals true

Make Integration

Use the HTTP module to add lightweight memory to scenarios.

1

Recall state at the start

Use an HTTP Make a request module pointed at /v1/recall.

2

Map the response body

Evaluate success and value in filters or routers. Do not rely on status code-based error handling.

3

Write state back

At the end of the scenario, store the new state with a POST module.

HTTP module configuration
Method: DELETE
URL: https://your-domain/v1/forget?key=hourly_lock&namespace=alerts
Headers:
  X-API-Key: {{connection.apiKey}}

Authentication

FlowState uses a single header-based API key for all protected endpoints.

Header

X-API-Key Exactly 8 characters matching [A-Z0-9]

Protected endpoints

  • POST /v1/store
  • GET /v1/recall
  • DELETE /v1/forget
  • GET /v1/health is the only endpoint without auth.
fetch('/v1/recall?key=last_run&namespace=invoice-sync', {
  headers: {
    'X-API-Key': 'K7M2X9BF'
  }
});

Limits & Validation

These are the rules your client should enforce before sending requests.

Field Rule
API KeyFixed 8 chars, [A-Z0-9]
key1-32 chars, [a-zA-Z0-9_-]
namespaceOptional, max 32 chars, [a-zA-Z0-9_-], defaults to _default
valueRequired, not null, max 1KB (1024 bytes)
ttlSeconds0-86400 seconds

Client-side guidance

If you build a reusable connector, validate these rules before calling FlowState so your automation users get immediate, readable feedback.

Examples

                
Common FlowState patterns for real automation systems.


                

                    

                        01
                        
Deduplication

                        
Question: Has this invoice already been processed?

                        
Store the external invoice ID after a successful run. Recall it on the next run to skip duplicates safely.

                    

                    

                        02
                        
Last Run Tracking

                        
Question: When did this workflow last execute?

                        
Store an ISO timestamp after completion and use it as the next polling cursor.

                    

                    

                        03
                        
Cross-workflow State

                        
Question: How do I pass data between different workflows?

                        
Write data in one workflow and recall it from another using a shared namespace and agreed key naming.

                    

                    

                        04
                        
Rate Limiting

                        
Question: How do I send max 1 notification per hour?

                        
Store a per-recipient lock with ttlSeconds: 3600 before sending the notification.

                    

                    

                        05
                        
State Machine

                        
Question: What state is this request in right now?

                        
Persist values like pending, approved, or rejected so different workflows can coordinate transitions.
