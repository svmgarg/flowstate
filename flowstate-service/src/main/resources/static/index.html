Workflow Memory API

Memory for your workflows

FlowState gives automation tools a simple REST memory layer so each workflow run can remember what happened before.

Built for Zapier, n8n, Make, and any custom automation that needs lightweight state without standing up a database.

Simple REST API

Store, recall, and forget values with predictable JSON responses that fit cleanly into HTTP nodes and webhook steps.

Automation Friendly

Designed for no-code and low-code tools that need persistence between workflow executions without extra infrastructure.

Predictable Contract

Every endpoint returns HTTP 200. Inspect success, message, and value in the JSON body to decide next steps.

How FlowState fits into a workflow

1
Write memory

Store a key when a workflow completes an important step.

2
Read memory later

Recall the key in the next run to avoid duplicate work or continue from the latest state.

3
Expire or delete

Use TTLs or explicit deletes to keep state fresh and scoped to the workflow lifetime.

Quick Start

From zero to workflow memory in three steps.

Get your API key

All endpoints except /v1/health require the X-API-Key header. Keys must be exactly 8 uppercase letters or digits.

To request an API key, contact the admin at svmgarg@gmail.com

Length: 8 Pattern: [A-Z0-9] Header: X-API-Key

Store a value

Send the key, value, TTL, and namespace. Namespaces are optional and default to _default.

curl -X POST https://your-domain/v1/store \
  -H "X-API-Key: K7M2X9BF" \
  -H "Content-Type: application/json" \
  -d '{"key": "last_run", "value": "2024-01-15T10:30:00Z", "ttlSeconds": 86400, "namespace": "invoice-sync"}'

Recall it later

Read the stored value in a future workflow run. Use the same key and namespace pair.

curl "https://your-domain/v1/recall?key=last_run&namespace=invoice-sync" \
  -H "X-API-Key: K7M2X9BF"

Important response behavior

FlowState always returns HTTP 200. Your workflow must check the JSON body and branch on success rather than HTTP status code.

POST

/v1/store

Persist a value for a key, optionally scoped by namespace and TTL.

Request body

keyRequired1-32 chars · [a-zA-Z0-9_-]
valueRequirednot null · max 1024 bytes
ttlSecondsOptional0-86400
namespaceOptionalmax 32 chars · defaults to _default

Auth

Requires X-API-Key with exactly 8 characters matching [A-Z0-9].

Success contract

The body includes success and message. Your workflow should treat success: false as a handled failure even though HTTP 200 is returned.

curl -X POST https://your-domain/v1/store \
  -H "X-API-Key: K7M2X9BF" \
  -H "Content-Type: application/json" \
  -d '{"key": "last_run", "value": "2024-01-15T10:30:00Z", "ttlSeconds": 86400, "namespace": "invoice-sync"}'
{
  "success": true,
  "message": "Value stored successfully"
}
POST

/v1/memory/store-if-absent

Atomic idempotency check — store a value only if the key doesn't already exist.

Request body

namespaceRequiredstring · logical grouping for keys
keyRequiredstring · 1-32 chars · [a-zA-Z0-9_-]
valueRequiredany valid JSON-serializable value · max 1024 bytes
ttlSecondsOptionalnumber · 0-86400 · defaults to 86400 (24 hours)

Headers

X-API-Key Required · 8 chars [A-Z0-9]
X-Workspace-Id Required · workspace identifier

Atomic behavior

Uses Redis SETNX under the hood — the check-and-store operation is a single atomic command. Zero race conditions, even under concurrent requests.

Unlike /store, this endpoint never overwrites existing data

If the key already exists, the existing value is preserved unchanged. The response tells you whether the key was new (existed: false) or already present (existed: true). This makes it perfect for idempotency checks where you need to answer: "Have I processed this event before?"

Request example

curl -X POST https://your-domain/v1/memory/store-if-absent \
  -H "X-API-Key: K7M2X9BF" \
  -H "X-Workspace-Id: ws_prod_01" \
  -H "Content-Type: application/json" \
  -d '{
    "namespace": "webhooks",
    "key": "evt_stripe_abc123",
    "value": "processed",
    "ttlSeconds": 86400
  }'

Response — New key stored

{
  "success": true,
  "message": "Stored (new key)",
  "existed": false
}

Response — Key already exists

{
  "success": true,
  "message": "Already exists (duplicate)",
  "existed": true
}

Use case: Deduplicating webhook deliveries

The problem

Payment providers (Stripe, PayPal, Razorpay) may deliver the same webhook event multiple times due to retries, network issues, or failovers. Without atomic deduplication, your workflow could process the same payment twice.

The solution

Call store-if-absent with the event ID as the key. If existed: true, skip processing — it's a duplicate. If existed: false, the key was stored atomically and you're safe to proceed.

// Webhook handler — atomic deduplication
const response = await fetch('/v1/memory/store-if-absent', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'X-API-Key': 'K7M2X9BF',
    'X-Workspace-Id': 'ws_prod_01'
  },
  body: JSON.stringify({
    namespace: 'webhooks',
    key: event.id,           // e.g. "evt_stripe_abc123"
    value: 'processed',
    ttlSeconds: 86400
  })
});

const result = await response.json();

if (result.existed) {
  console.log('Duplicate webhook — skipping');
  return { status: 200 };
}

// First time seeing this event — safe to process
await processPayment(event);
return { status: 200 };

Why not use /store + /recall?

A two-step recall-then-store pattern has a race condition: two concurrent requests could both see "key not found" and both proceed. store-if-absent eliminates this by combining the check and write into a single atomic operation (Redis SETNX). There is no window where two callers can both "win."

/v1/recall

Fetch a value that was stored earlier.

Query parameters

keyRequiredsame validation as store
namespaceOptionaldefaults to _default

Response

When found, the response includes value. Missing or invalid requests still come back as HTTP 200 with success: false.

  -H "X-API-Key: K7M2X9BF"
{
  "success": true,
  "message": "Value found",
  "value": "2024-01-15T10:30:00Z"
}

Check service availability without authentication.

No auth required

Zapier Integration

Give your Zaps memory. Prevent duplicate emails, remember where you left off, and share state between runs — using just a Webhook step.

The Problem

Zapier workflows are stateless — each run starts from scratch. This causes real problems:

  • Duplicate actions — Stripe sends the same webhook twice → your Zap creates two orders
  • Re-processing old data — a scheduled Zap processes all 500 invoices every hour instead of just new ones
  • Double emails — a user submits a form, refreshes, and gets two welcome emails

FlowState fixes this — it stores your workflow's state so your Zap knows what it already did.

How It Works

Add a single Webhooks by Zapier step to your Zap. FlowState gives you two patterns:

🔁 Checkpointing

"Remember where I left off"

Store the last processed record ID after each run. Next run reads it and skips everything before it.

Examples:

  • Sync only new Shopify orders since last run
  • Process CRM contacts added after last sync
  • Resume a multi-page API import from where it stopped

🛡️ Deduplication

"Have I already handled this?"

Before taking action, ask FlowState: "have I seen this event ID before?" If yes, skip. If no, proceed and mark it as seen — in one atomic call.

Examples:

  • Prevent duplicate Stripe webhook processing
  • Send only one welcome email per signup
  • Avoid creating duplicate tickets from retried form submissions

Example: Stop Duplicate Webhooks

Scenario: Stripe fires a payment_intent.succeeded webhook. Sometimes it arrives twice. You don't want to create two orders.

Solution: Add one Webhook step at the start of your Zap:

Method: POST
URL: https://your-domain/v1/memory/store-if-absent
Headers:
  X-API-Key: YOUR_KEY
  Content-Type: application/json
Body:
{
  "key": "evt_stripe_pi_3ABC123",
  "value": "processed",
  "ttlSeconds": 86400,
  "namespace": "stripe-dedup"
}

Response tells you what to do:

Response Meaning Action
{ "existed": false } First time seeing this event Continue your Zap → create the order
{ "existed": true } Duplicate event Use a Filter step to stop the Zap

Example: Process Only New Records

Scenario: Every hour, your Zap syncs invoices from an accounting tool. You only want new ones since the last run.

Step 1 — Read the last checkpoint:

Method: GET
URL: https://your-domain/v1/recall?key=last_invoice_id&namespace=invoice-sync
Headers:
  X-API-Key: YOUR_KEY

If a value exists, use it in a Filter to skip invoices you've already processed.

Step 2 — Save the new checkpoint after success:

Method: POST
URL: https://your-domain/v1/store
Headers:
  X-API-Key: YOUR_KEY
  Content-Type: application/json
Body:
{
  "key": "last_invoice_id",
  "value": "INV-2026-0542",
  "ttlSeconds": 86400,
  "namespace": "invoice-sync"
}

Next run reads INV-2026-0542 and starts from there — no re-processing.

More Use Cases

💳 Payment Webhooks

Stripe, Razorpay, PayPal all retry webhooks. Use the event ID as key to catch duplicates.

📧 One Email Per User

Send only one welcome email per signup. Key: welcome_user_42 with 24h TTL.

🛒 Order Dedup

Prevent double orders when Zapier retries. Key: the order or cart ID.

📊 Incremental Syncs

Sync CRM contacts or invoices since last run. Store the last processed timestamp or ID.

⚠️ Important for Zapier Users

FlowState always returns HTTP 200 — even for errors. In your Zap, use a Filter or Paths step to check the success field in the JSON body (not the HTTP status code).

n8n Integration

Use HTTP Request nodes to persist state between executions.

Recommended node setup

  • Use HTTP Request nodes for FlowState calls.
  • Set headers explicitly, including X-API-Key.
  • Read {{$json.success}} in IF nodes because the transport status is always 200.

Best fit scenarios

  • Skipping records already processed
  • Sharing data across separate n8n workflows

Authentication

FlowState uses a single header-based API key for all protected endpoints.

Header

X-API-Key Exactly 8 characters matching [A-Z0-9]

Protected endpoints

  • POST /v1/store
  • POST /v1/memory/store-if-absent
  • GET /v1/recall
  • DELETE /v1/forget
  • GET /v1/health is the only endpoint without auth.
fetch('/v1/recall?key=last_run&namespace=invoice-sync', {
  headers: {
    'X-API-Key': 'K7M2X9BF'
  }
});

Examples

                
Common FlowState patterns for real automation systems.


                

                    

                        01
                        
Deduplication

                        
Question: Has this invoice already been processed?

                        
Store the external invoice ID after a successful run. Recall it on the next run to skip duplicates safely.

                    

                    

                        02
                        
Last Run Tracking

                        
Question: When did this workflow last execute?

                        
Store an ISO timestamp after completion and use it as the next polling cursor.

                    

                    

                        03
                        
Cross-workflow State

                        
Question: How do I pass data between different workflows?

                        
Write data in one workflow and recall it from another using a shared namespace and agreed key naming.

                    

                    

                        04
                        
Rate Limiting

                        
Question: How do I send max 1 notification per hour?

                        
Store a per-recipient lock with ttlSeconds: 3600 before sending the notification.

                    

                    

                        05
                        
State Machine

                        
Question: What state is this request in right now?

                        
Persist values like pending, approved, or rejected so different workflows can coordinate transitions.
